# 项目说明

## 宪法定位
本文件为 OneRss 项目“宪法”，用于统一团队在研发、协作与交付过程中的共同约束。  
本宪法描述的是**原则与边界**，不描述具体实现细节；如与临时实现方案冲突，以本宪法为准。

## 技术栈约束
- 项目采用 `Expo + Supabase` 作为基础技术栈。
- 技术选型遵循“稳定、可维护、可审计”原则，不追逐实验性方案。
- 新增技术能力必须与现有栈兼容，避免引入破坏性复杂度。
- 涉及平台能力时优先选择跨端一致方案，确保移动端体验一致性。

## 包管理规范
1. 仅使用稳定版本依赖，禁止引入 beta、alpha、rc、canary 等预发布版本。  
2. 新增依赖前必须评估必要性，能复用现有能力则不新增。  
3. 依赖升级采用“可回滚”策略，升级后需验证核心流程。  
4. 锁文件必须纳入版本管理，确保团队环境一致。  
5. 禁止同时混用多套包管理体系，统一使用项目既定包管理方式。

## 工程化规范
在编写代码时，必须遵循以下原则：
1. 严禁重复：任何重复逻辑都必须抽象为可复用单元。  
2. 先查后写：实现前先检查项目中是否已有类似能力。  
3. 模块化设计：按职责拆分模块，降低耦合。  
4. 统一接口：参数格式、返回结构、错误语义保持一致。  
5. 配置集中管理：所有常量与配置集中管理，禁止散落。  
6. 优先封装：相关能力在明确边界内封装，不向外泄漏内部细节。  
7. 环境变量管理：敏感配置使用 `.env` 管理，禁止硬编码。  
8. 可测试性优先：代码结构必须便于单元测试与集成测试。

## 开发方式
严格执行 TDD（红-绿-重构）：
1. 先分析需求并拆分测试点。  
2. 先编写失败测试（Red）。  
3. 再写最小实现使测试通过（Green）。  
4. 最后在测试保护下重构（Refactor）。  
5. 每次提交前必须保证核心测试通过，禁止“先合并后修复”。

## 沟通规范
1. 团队沟通统一使用中文（含评审意见与注释说明）；面向用户的产品文案与错误提示使用英文。  
2. 需求存在歧义、缺失或冲突时，必须先澄清再实施。  
3. 沟通表达需清晰、简洁、技术准确，避免模糊结论。  
4. 对关键决策需记录“背景-选择-影响”，保证可追溯。  
5. 当前任务完成后，必须给出下一步可执行建议。

## 执行任务规范
1. 复杂问题必须先拆解，再执行；优先小步快跑与阶段性验证。  
2. 涉及 UI 的任务必须进行视觉回归核验，确保与原型一致。  
3. 搜索与批处理操作应显式忽略无关目录（如依赖目录与构建产物）。  
4. 任何高风险操作（数据迁移、权限变更、批量删除）必须先评审并备份。  
5. 自动化优先：重复性流程应脚本化并纳入标准流程。

## 安全与隐私要求
1. 最小权限原则：每个角色仅拥有完成职责所需的最小访问权限。  
2. 数据隔离原则：用户只能访问其本人数据，禁止越权读取与修改。  
3. 默认安全：所有接口默认鉴权，显式放开必须有审计依据。  
4. 密钥与凭据零硬编码：密钥、令牌、连接串必须通过安全配置注入。  
5. 隐私最小化：仅采集业务必需数据，禁止采集无关敏感信息。  
6. 日志脱敏：日志中禁止输出敏感字段与可识别个人信息。  
7. 可审计性：关键安全事件必须可追踪、可审计、可告警。  
8. 合规删除：用户触发删除或注销时，必须支持按策略清理个人数据。

## 一致性要求
1. 体验一致：跨页面的导航、交互反馈、状态呈现保持一致。  
2. 视觉一致：颜色、字体、间距、圆角、图标语义遵循统一设计令牌。  
3. 领域一致：同一业务概念使用统一命名，禁止同义多名。  
4. 行为一致：同类操作在不同入口应有相同结果与错误语义。  
5. 文档一致：需求、接口、测试与实现需同步更新，保持对齐。

## 质量基线与测试要求
- 不允许不安全类型捷径与静态检查规避手段。  
- 代码需通过基础静态检查与构建检查。  
- 提交前必须执行标准化检查流程（格式、Lint、测试、类型检查）。  
- CI 必须包含自动化质量门禁与安全扫描，未通过不得合并。  
- 关键路径（登录、订阅、阅读、离线）必须具备回归测试覆盖。

## TypeScript 约束
1. 禁止滥用 `any`、`unknown`、类型断言绕过检查。  
2. 以精确类型建模业务语义，避免宽泛类型污染边界。  
3. 公共接口必须显式声明输入输出类型。  
4. 任务完成后必须执行 TypeScript 类型检查，保证零阻断错误。

## 变更与治理
1. 本宪法变更需通过评审并记录版本。  
2. 新增规范不得与既有条款冲突；如冲突，需先完成条款修订。  
3. 所有成员对本宪法负有遵循责任，例外必须说明并获批准。
